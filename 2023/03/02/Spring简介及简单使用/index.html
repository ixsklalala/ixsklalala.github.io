2 IoC、DI和AOP思想提出

  • IoC：Inversion of Control，控制反转，强调的是原来在程序中创建Bean的权利反转给第三方
  • DI：Dependency Injection，依赖注入，强调的是Bean之前的关系，这种关系由第三方复制去设置
  • AOP：Aspect Oriented Programming，面向切面编程，功能的横向抽取，主要的实现方式就是Proxy

3 Spring框架的诞生

3.1 Spring概述

Spring是一个开源的轻量级Java开发应用框架，可以简化企业级应用开发。Spring解决了开发者在JavaEE开发中遇到的许多常见的问题，提供了功能强大的IoC、AOP及Web MVC等工。是当前企业中Java开发几乎不可缺少的框架之一。Spring的生态极其完善，不管是Spring哪个领域的解决方案都是依附于Spring基础框架的
Spring的官网：www.spring.io
Spring框架历史：

  • Jsp默默抗下所有
  • MVC+三层架构分工明确，但开发成本极其高
  • EJB重量级框架出现，走出一个困境，又走进另一个困境
  • Spring春天到来，随之，SSH风生水起
  • Spring稳住地位，SSM开始上位
  • Spring本着“拿来主义”的思维快速发展，生态不断健全
  • SpringBoot又一里程碑崛起，把“约定大于配置”思想玩的炉火纯青
  • SpringCloud打包了微服务众多解决方案，应对互联网项目更加简单

3.2 Spring技术栈

  • Data Access：数据访问
  • Data Integration：数据集成
  • Web：Web开发
  • AOP：面向切面变成
  • Aspects：AOP思想实现
  • Core Container：核心容器
  • Test：单元测试与集成测试

4.2 ApplicationContext

ApplicationContext成为Spring容器，内部封装了BeanFactory，比BeanFactory功能更丰富更强大，使用ApplicationContext进行开发时，xml配置文件的名称习惯写成applicationContext.xml

4.3 BeanFactory和ApplicationContext的关系

  • BeanFactory是Spring的早期接口，称为Spring的Bean工厂，ApplicationContext是后期更高级的接口，称之为Spring容器
  • ApplicationContext在BeanFactory基础上对功能进行了扩展，如：监听功能、国际化功能等。BeanFactory的API更偏向底层，ApplicationContext的API大多数是对这些底层API的封装
  • Bean创建的主要逻辑和功能都被封装在BeanFactory中，ApplicationContext不仅继承了BeanFactory，而且ApplicationContext内部还维护者BeanFactory的引用，所以，ApplicationContext与BeanFactory既有继承关系，又有融合关系
  • Bean的初始化时机不同，原始BeanFactory是在首次getBean时才进行Bean的创建（懒汉模式），而ApplicationContext则是配置文件加载，容器一创建就将Bean都实例化并完成初始化（饿汉模式）

4.4 BeanFactory和ApplicationContext的继承体系

  • BeanFactory的继承体系
    BeanFactory是核心接口，项目运行过程中肯定有具体实现参与，这个具体实现就是DefaultListableBeanFactory，而ApplicationContext内部维护的BeanFactory的实现类也是它


  • ApplicationContext的继承体系（仅导入spring-context依赖时）

    ClassPathXmlApplicationContext：加载类路径下的xml配置的ApplicationContext
    FileSystemXmlApplicationContext：加载磁盘路径下的xml配置的ApplicationContext
    AnnotationConfigApplicationContext：加载注解配置类的ApplicationContext

5.1.3 Bean的范围配置

默认情况下，单纯的Spring环境Bean的作用范围有两个：Singleton和Prototype

5.1.3.1 singleton

singleton：单例，默认值，Spring容器创建的时候，就会进行Bean的实例化，并存储到容器内部的单例池中，每次getBean时都是从单例池中获取相同的Bean实例

5.1.3.2 prototype

prototype：原型，Spring容器初始化时不会创建Bean实例，当调用getBean时才会实例化Bean，每次getBean都会创建一个新的Bean实例


从上图可以看出，当scope设置为prototype时，通过getBean获取对象时，会创建一个新对象，并不会存入singletonObjects中
但是scope只有这两种取值吗？其实并不是，当前的取值仅针对spring-context基础环境，如果引入spring-mvc，还会有两种取值request和session，如果配置request，则对象作用在request域，配置为session，对象作用在session域

5.1.4 Bean的延迟加载

当lazy-init设置为true时为延迟加载，也就是当Spring容器创建的时候，不会立即创建Bean实例，等待用到时再创建Bean实例并存储到单例池中，后续再使用该Bean时，直接从单例池中获取即可。注意该配置仅针对ApplicationContext有效，对BeanFactory是无效的
